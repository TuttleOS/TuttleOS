# Tuttle OS — UI Design Decisions (running log)

Established with Michael across mockup sessions, 2026-07-13. These are standing rules for ALL workspaces.

## Architecture
- One app, role-based workspaces: Intake / Case Manager / Litigation Paralegal / Demand Writer (Kate) / Lien-Disbursement (Emily) / Senior Paralegal Reviewer (Daniel) / Owner (Michael).
- Security lives in the database (RLS), never the UI. Locked nav items shown with 🔒 + tooltip naming the enforcing tier.
- Intake's narrow world is DATABASE-ENFORCED (not cosmetic): blocked from medical, litigation, resolution, liens, insurance, property, finance.

## Universal rules
0. **Global search bar on EVERY screen, no exceptions** — sticky top bar, always visible. Autocompletes client names as you type; every result shows the DOI (crash date) so a client with multiple cases is disambiguated (CasePeer convention: "Last, First - date"). Searches name, phone, email, claim #, cause #, CasePeer id — results respect database permissions. Matching is PARTIAL everywhere: any substring of a name matches ('kafo' → Okafor), and phone matching is digit-normalized so formatting never matters ('2105550119', '555-0119', or just '0119' all find (210) 555-0119). The schema's phone_e164 column is the real-build backing for this. Pressing Enter WITHOUT selecting a dropdown item opens a full SEARCH RESULTS PAGE with matches grouped by category (Cases & Clients · Providers · Adjusters & Carriers · Defense Counsel · Mediators · Depositions · Notes · Contact log · Approvals · Staff — per workspace). Arrow keys + Enter (or click) still open a specific hit directly; the dropdown always ends with a "See all results" row.
1. **Every case name is a hyperlink** to the case — in queues, tasks, calls, reports, everywhere.
2. **Every provider name is a hyperlink** → contact card; its phone dials RingCentral **bound to the case** and opens the provider contact log ("initiated" → complete-the-log flow).
3. **Case header always answers "how do I reach this person"**: phone (tel:/RingCentral, auto-logged), email (mailto + copy button for cc'ing), DOB with computed age. No scrolling.
4. **Adjusters are a roster per claim** (BI/PD/PIP/UM-UIM/litigation/supervisor functions), each with own phone+ext/email links; calls bind matter+claim+adjuster. 💬 history per adjuster = firm-wide cross-case notes (insurance.adjuster_note).
5. **Completed sections roll up**: done checklist items behind "▸ show N completed"; fully complete sections collapse to a single ✓ line, expandable. Screen space goes to what's LEFT.
6. **Document-backed tasks complete themselves** when the document is logged (completion_method=system). Manual completion of a doc-backed task = OVERRIDE requiring a reason; overrides badge on the task and feed Michael's v_task_override_patterns report.
7. **Tasks complete anywhere** (My Tasks or case page) — same record, same rules (gating honored), audit-stamped, reopen allowed.
7a. **EVERY role gets a My Tasks list + anyone can hand anyone a task (Michael 2026-07-13)**: the paralegal workspace now has My Tasks grouped BY LITIGATION STAGE (Filing & Service · Answers & Disclosures · Written Discovery · Depositions · Mediation · Trial Prep · General). A universal '➕ Task for a co-worker (or me)' form (My Tasks tab + every case page, case preselected) writes workflow.task with owner = assignee, created_by = sender; it lands in the assignee's My Tasks with a 'from X' tag and the sender sees completion in their My Activity. NOT redundant with deadlines — **deadlines say WHEN, task lists say WHO does WHAT.**
7b-i. **DYNAMIC TASK CHAINS — event-driven state machine (Michael 2026-07-14)**: litigation task lists are NOT a static 100-item dump — each task's completion is the EVENT that spawns the next, so the list only ever shows work that is actionable NOW ("tasks that are not even ready to be worked on are not cluttering up a task list") and the case can never stall silently (something is always pending, aging, owned). THE SERVICE CHAIN (built + tested end-to-end): case FILED → 'Request citation — [defendant]' → done → 'Confirm citation RECEIVED (ages until in hand)' → done → 'Forward citation to process server' → done → '5-day follow-up' that RESPAWNS every 5 days (follow-up #N) until the RETURN OF SERVICE is entered on the case — the data entry kills the loop (auto-complete, method=system) and spawns 'Confirm answer filed — [defendant] (TRCP 99 due <date>)', which the ANSWER ENTRY auto-completes, handing off to the disclosures template. Design rules: (a) chains instantiate PER DEFENDANT with the name in the title; (b) HUMAN steps complete by checkbox, RECORD-PROVABLE steps complete by the data entry itself (rule 6 applied to litigation — never ask a human to confirm what the record already proves); (c) loops respawn with attempt counters and die on their terminal data event; (d) every spawn/kill logs to My Activity. DB: workflow.task already supports it (trigger_source, recurring_interval); future v2.7 = ref.task_chain_step (chain, step, title_tpl, due_offset, completes_by manual|data_event, respawn_days, next_step) + completion/data-event triggers so chains run in the database like the contract checklist. **MULTI-DEFENDANT COVERAGE (Michael 2026-07-14)**: filing spawns one chain PER DEFENDANT (2 defendants = 2 independent chains, each advancing at its own pace), and a per-case **service coverage board** at the top of My Tasks audits the roster: every defendant shows its current chain position ('step 2 of 5 — confirm citation received'); a defendant whose chain died mid-stream flags '⚠ CHAIN STALLED — last completed step N' with a one-click '▶ start next link'; a defendant with NO chain at all (e.g. added by amended petition) flags '🛑 NO CHAIN — citation never requested'. The board is the thoroughness guarantee: a case is not 'in service' until EVERY defendant reaches ✓. Chains to define next: discovery-response chain, depo-scheduling chain, mediation chain, trial-prep chain.
7b. **Litigation stage task TEMPLATES (Michael 2026-07-13)**: stage events generate the stage's work items automatically, same engine as the sign-up checklist (workflow.task + trigger_source): filing → service tasks; FIRST answer → disclosure-prep task (due ~2 weeks before the 194.2(a) date); DCO w/ mediation deadline → mediator-selection task; depo scheduled → its readiness checklist; trial setting <120d → trial-prep tasks. Guarded against double-fire per case. Embedded checklists (depo readiness, trial prep) are the same concept — templates formalize them. DB needs no change (workflow.task supports it all); a future v2.7 could add ref.task_template + stage-event triggers to move generation into the database like the contract checklist.
8. **Every case carries a CM AND a paralegal (Michael 2026-07-13)** — assigned from day one (workflow §2). Missing assignments FLAG (never block): red 'UNASSIGNED' on the case, row in core.v_matter_role_gaps (v2.1 §N) on the owner dashboard. **The assigned CM and paralegal show on EVERY screen for EVERY employee** — case headers, caseload rows, dashboards, everywhere a case appears.
9. **Cross-workspace views — WORKSPACE SWITCHER (Michael 2026-07-13, chose from 3 options)**: paralegals get the FULL Case Manager workspace — every tab and every function (provider calls, treatment logging, PD entry, checklists, adjuster roster), not a read-only dashboard — via a persistent '🔁 Case Manager view' toggle in the top bar. Switching swaps the entire workspace; an amber banner keeps identity explicit ('you are <paralegal> — everything audits under YOUR name') with a one-click '← Back to Paralegal workspace'. ONE CM workspace codebase serves CMs and paralegals — no duplication, no drift; the paralegal DB tier already permits everything, so this is pure UI composition (the toggle appears only for users holding the litigation_paralegal role). Rejected alternatives: nesting the CM workspace inside a paralegal tab (tabs-within-tabs), and a combined two-section sidebar. The reverse direction is NOT symmetric: the CM gets only the LIMITED milestone paralegal view — filed, discovery phase, depos being scheduled, mediation set, trial date — enough for "what's happening with my lawsuit?" on a client call; no rule citations, request-level discovery, or diligence logs. (Further parameters TBD — Michael.)
10. **"My Activity" tab for EVERY user (Michael 2026-07-13)** — each user's own slice of the audit stream, newest first: tasks, calls, notes, PD updates, LOR dates, overrides, approvals. Top entry highlighted "⟵ you left off here" with a Resume button that opens the case — for coming back after a phone call or meeting. No extra logging required; the audit layer already attributes everything.
11. **Client-call follow-up dispatcher — EVERY user, EVERY case (Michael 2026-07-13)**: a prominent '📞 Client called — create follow-up' button on every case header, regardless of whose case it is. One form: what the client needs* · route to primary owner (CM / paralegal / attorney, prefilled from the case team) · due (today/tomorrow/this week) · urgent flag. The PRIMARY owns the task; the OTHER teammate automatically sees it in their My Tasks tagged 'backup — primary: X' so either can follow up. Saving also logs the call to communication_log, drops a case note ('CLIENT CALLED: …'), and lands in the dispatcher's My Activity. Empty request refused — the request text IS the task. (Open item: intake answering about an existing case can dispatch via search without opening restricted case data.)
12. **Status colors always ship icon + label**, never color alone. JX (red) = jurisdictional/missable; COURT ORDER (blue) = supersedes rule defaults.
12a. **No dead-end status — every status indicator is a LINK to where the work happens (Michael 2026-07-13: "carried over to all screens for everyone")**: CM header badges ('PD: In Progress ↓', 'Auths: Outstanding ↓') jump to and expand their section (Focus-view aware — a collapsed card expands first, then scrolls into view with a highlight flash); paralegal timeline nodes jump to the section they describe (Service→Defendants, Mediation→Mediation card, 18.001→Deadlines); intake gate items ('✕ 2 of 6 outstanding — click to fix: Phone, Email') scroll to and focus the exact form field; owner tiles navigate to their queues. When the target section doesn't exist on a file, the click explains why instead of dying (e.g. 'Lien detail lives in the Lien & Disbursement workspace — restricted tier'). If a user can SEE a status, they can CLICK it to act on it — applies to every workspace built from here on.
13. **Multi-plaintiff / companion cases (Michael 2026-07-13)** — one crash, multiple clients = core.incident_group; EACH client is their OWN matter (own SOL, own medicals, own negotiation, own settlement). UI: 👪 'N of M · same crash' badge on every caseload row; a **companion strip** at the top of each member's case page — all members linked (current file highlighted), per-member role (driver/passenger/minor + next friend), conflict-waiver chip (green cleared w/ detail on hover, red pending = NO cross-file sharing; representation_link), copy-sharing chip, and the SHARED-vs-NEVER-SHARED ledger: police report = one portal pull filed to all; same at-fault policy = **aggregate-limits warning** ('3 claimants share the $60k aggregate — allocation & sequencing = ATTORNEY decision, demands tracked together'); PD only on the vehicle owner's file; medicals/treatment/negotiation/settlement strictly per client. **Minors**: MINOR badge + next friend in header and caseload, SOL shows '⚠ ATTORNEY-VERIFY' (never a computed number), friendly-suit/court-approval flag at settlement. **In litigation** (paralegal view): one cause number, N plaintiff-side lit_party rows — 'Plaintiffs (N)' block on the court case card with companion links; court deadlines (DCO/discovery/trial) SHARED, § 18.001 affidavits + damages + settlement PER PLAINTIFF; 👪 badge on the litigation caseload. Search disambiguates same-surname members by DOI (already universal rule 0).
14. **Focus view — progressive disclosure by default (Michael 2026-07-13: anti-clutter)**: production case pages open in FOCUS — a '🎯 Needs you now' strip (overdue/red items, due calls, missing approvals, open checklist count, companion links) followed by every card COLLAPSED to its one-line header. Cards stay open automatically when HOT (red/serious status or an open form); click any header to expand/collapse; '🗂 Full view' toggle restores the everything-open page. Principle: everything reachable, only the actionable visible — the hierarchy is act-first, reference-on-demand. Built in the CM workspace; the pattern applies to every workspace as they're productionized.
15. **Data capture follows the workflow (Michael 2026-07-13)**: every UI input maps 1:1 to a typed database surface — provider call form → medical.provider_contact_log; attempt log → service diligence; answer entry → lit_party (fires the 18.001 engine); DCO form → litigation.scheduling_order (fires the deadline engine). Three invariants: (a) forms live WHERE the work happens (log-call on the provider row, serve on the defendant box) — never a generic 'edit record' screen; (b) every write is role-gated and validated AT THE DATABASE (RLS + constraints + mandatory audit actor) — the UI can't grant what the DB forbids; (c) every capture feeds the machine — one typed date becomes deadlines, calendar entries, timeline nodes, and dashboard flags with zero re-typing. The remaining workspaces (demand writer, lien/disbursement, senior reviewer) follow the same rule: their forms ARE their workflow.
16. **Every displayed date shows day, month, AND year (Michael 2026-07-13)** — no year-less dates anywhere in the UI: 'Jul 13, 2026', never 'Jul 13'. Applies to everything — due dates, deadlines, badges ('✓ cervical (May 28, 2026)'), timestamps ('Jul 13, 2026, 8:42 AM'), activity logs, treatment logs, call queues. All toLocaleDateString calls must include year:'numeric'. A PI file lives for years; 'Apr 2' turns ambiguous the moment a case crosses a calendar year.

## Sign-up checklist (9 tasks, DB-generated at signature, bilingual double-fire guarded)
HIPAA auths · Open DINSCO claim · Send DINSCO LOR (gated on claim) · Open PINSCO claim OR document refusal · Send PINSCO LOR (waived on refusal) · Obtain PINSCO Dec Sheet (waived ONLY on documented refusal) · Police report · Case Profile · SOL confirm.
**LOR tasks: generated ≠ SENT.** The completion proof for both LOR tasks is insurance.claim.lor_sent — the date the letter ACTUALLY went out, entered by the user on the claim card (required; empty date refused). Document generation alone never completes an LOR task.
**Police report: obtained via portal, not ordered.** Task = 'Obtain police / crash report — portal, retry until in file', HIGH priority; completes only when the CR-3 document is saved to the file. Urgency stays on until then.
PINSCO refusal: documented fact (amber "!", not green ✓) → coverage_assessment.pinsco_permission_to_use=No + note; feeds Level inputs.

## Case Manager workspace
- Tabs: My Caseload / Provider Calls / My Tasks (+ locked Financials, Discovery Content).
- Caseload row: client (+TBI badge), stage, age/days-in-stage, Level, treatment, PD, SOL countdown, red flags.
- Case page cards: contact strip → badges wall → checklist → property damage → treatment/clinical events (or provider picker w/ LOP flag for attorney) → records tracking → demand readiness → negotiation → Insurance & Claims (adjuster roster + empty-state "open claim" buttons that complete checklist tasks) → notes / next actions.
- **Treatment = ALL providers, with coverage boxes (Michael 2026-07-13)**: one treatment_episode per provider; categories Ambulance/EMS · Hospital/ER · Urgent care · Chiro · PT · Imaging · Pain mgmt · Surgical · Other. EVERY box must be answered — add provider(s) (directory-datalist or new, optional LOP flag → attorney queue) or explicitly N/A (undoable). Unanswered boxes show red ☐ + an 'N unchecked' badge; missed ambulance bills become surprise liens. Episodes table shows type/provider/status/balance; PONS anchor starred; clinical-events log lives inside the card.
- **Repeat visits (Michael 2026-07-13)**: NO new structure — same episode, schema's visit_count/first_visit/last_visit. '+ visit' link on singular-event rows (ER/urgent/ambulance) logs a return: badge 'N visits', clinical event ('Return visit #N' — damages narrative), note, expect-another-bill reminder. THE KEY RULE: if a records request predates the new visit, a red '⚠ records supplement needed' flag fires — the prior request's date range misses the visit. Records themselves remain the source of truth for visit detail once they arrive; this is only the early-warning notation.
- **New-provider intake (Michael 2026-07-13)**: typing an unknown name in a coverage box opens the FULL provider record before the episode starts: name*, provider_type, phone*, fax, records method (fax/portal/email/mail/Datavant-Ciox) + records contact, accepts-LOP. ADDRESSES ARE THREE DIFFERENT THINGS: 🏢 business address* · 💵 payment/remit address ("same as business" checked by default; unchecking requires the lockbox/PO Box — where checks ACTUALLY go) · 📍 treatment locations (0..N labeled addresses where the client is seen; business address assumed if none). Each stored as its own labeled contact_point row on the provider org. Known-directory names still one-click.
- **Provider call input form (Michael 2026-07-13)** — 'Log call' expands an inline form mirroring medical.provider_contact_log: reached? (no-answer → short retry ~2d, NOT +14d) · treatment status (on track / gap concern / noncompliant / complete) · approx balance · next appointment · clinical-event checkboxes (MRI referral+where, MRI done, TPI, RFA, surgical referral) — **ESI and MBB are SEPARATE procedures, each with multi-region checkboxes (lumbar/cervical/thoracic/other)**: one visit can hit multiple body parts and each procedure+region logs as its own clinical event ('ESI — lumbar', 'ESI — cervical', 'MBB — lumbar'); 'other' region requires a typed body part (ESI/MBB/MRI all have the text box).
  **Treatment log (Michael 2026-07-13)**: every call's procedures append to a per-case treatment log (date · procedure · body part · provider · cost) with a running cost-to-date (sum of episode balances). **MRIs are once-per-body-part**: MRI-done is tracked per part (lumbar/cervical/thoracic/brain/other+text); parts already done render LOCKED in future call forms ('✓ cervical (May 28, 2026)') so staff never re-ask — multiple parts may share a date or not, and a repeat on the same part needs a documented reason.
  **Dates on everything the provider can tell us (Michael 2026-07-13)**: the call form captures 'date client was seen' (this report) and 'initial visit date' (input only when not already on file — locked '✓ on file — Apr 14, 2026' once known; sets episode first_visit/last_visit). Checking ANY procedure box (ESI/MBB region, MRI part) reveals an optional per-procedure date field; provided dates flow into the treatment log as the procedure's real date, undated ones log as 'Jul 13, 2026 (rptd)' — reported-on date, actual date unknown. Locked done-MRI badges show the procedure's real date · note. Saving snapshots the balance onto the episode, pushes clinical events + a note onto the case, attaches RingCentral duration when the call started from the provider card, self-schedules +14d, and gap/noncompliant ESCALATES immediately (case flag + Michael's dashboard). Provider click = contact card w/ case-bound dialing.
- **PD input flow (Michael 2026-07-13)**: Step 1 = vehicle facts (year/make/model, drivable, CURRENT LOCATION required — tow/storage yard starts the storage clock) with SOURCE recorded (client call and/or CR-3; the crash report confirms VIN/plate when it posts). Step 2 = ROUTE the PD claim: DINSCO PD (no deductible, needs liability acceptance) vs PINSCO collision (faster, deductible now + subro later) — creates the pd_dinsco/pd_pinsco insurance claim and ties property.pd_claim to it. A documented PINSCO refusal DISABLES the PINSCO route with a warning. Ongoing changes via ✎ update edit mode on the card — every save stamps last_touch (PD Aging reset) and logs a note.
- PD card mirrors property.pd_claim: location (storage-accruing warning), repair/total (disputes highlighted), valuation, lienholder, rental/LOU, DV, deductible; demand_blocker callout at demand stage; resolved → collapsed line; N/A → no card.

## Litigation Paralegal workspace
- **Calendar sync (Michael 2026-07-13)**: EVERY deadline row mirrors to the firm calendar (Outlook or Google — real build: Microsoft Graph / Google Calendar API). Adds, date moves, and vacaturs push automatically; nothing is calendared by hand twice. Overdue items pin red at the top of every list until resolved — they never age off silently.
- **Service → answer → DCO pipeline (Michael 2026-07-13)** — the litigation deadline engine:
  0. **Attempt logging** ('+ Log service attempt' on each unserved defendant, and a jump-link on every Service Incomplete row): date* + time* · address attempted* · attempted by* · outcome (no answer / moved / bad address / evading / occupant / no access / SERVED) · what happened* ('a reviewing judge reads THIS') · photo/evidence-saved notation · next-step planned (revises the case's next-step line). Attempts auto-number ('3rd attempt — …'), prepend newest-first, timestamp day-month-year + time, and land in My Activity. Same-day logging, every time — the log IS the due-diligence tolling record and the exhibit to a Rule 106 motion. Outcome = SERVED hands off directly to the return-of-service form (step 1) so the answer deadline computes and calendars.
  1. **Return of service entry** ('✓ SERVED' button on each unserved defendant): date served* · method (personal/certified/106/SoS) · served by/on · return-saved-to-file* gate (and note the CLERK filing date — TRCP 107(h)'s 10-day default clock runs from filing). Saving auto-computes the **answer deadline — TRCP 99, 10:00 AM Monday next after expiration of 20 days after service** — dockets it on the case, and pushes it to the firm calendar.
  2. **Overdue watch**: deadline passes with no answer entered → 🔔 ANSWER OVERDUE flags fire everywhere — red badge on the defendant, red callout on the case (default-judgment opportunity, with the caution that the answer may have arrived by mail unlogged — check the clerk's site first), dashboard tile, and Michael's dashboard. Rows stay red until an answer is entered.
  3. **Answer entry** ('📄 Answer filed' button): filed date* · answering counsel (creates/links the defense-counsel contact) · answer-doc-saved* gate. Saving vacates the TRCP 99 deadline and populates the FULL post-answer deadline set (Michael 2026-07-13):
     - **EVERY answer** (per-defendant clocks): ⚠ serve § 18.001 affidavits as to that defendant — EARLIER of 90 days after ITS answer or the P expert-designation deadline (CPRC 18.001(d); evidence-critical; HB 4595 anchor ATTORNEY-VERIFY) · counter-affidavit window closes at 120 days (18.001(e) — defense's deadline, watched: silence locks in our affidavits).
     - **FIRST answer only** (case-wide Level 2 set): initial disclosures 30 days, both sides (TRCP 194.2(a), JX) · Level 2 discovery period closes — PROVISIONAL (TRCP 190.3(b): computed 9 months after the first discovery anchor, recomputes as real anchors land, JX, `supersedable`) · Level 2 trial target at 18 months (Rules of Jud. Admin. 6.1 / DWOP watch, `supersedable`).
     - **Earlier-of enforcement**: when a DCO later sets P expert designation BEFORE a pending 18.001 90-day date, the affidavit deadline MOVES EARLIER automatically (annotated 'DCO expert designation controls'); the provisional L2 close and trial target vacate outright (court order beats rule). All computed rule dates are attorney-verify.
  4. **Scheduling order entry** (📋 button on the court case card; '✎ amended order' link once one exists): order signed date* · order-saved-to-file* gate · 11 standard DCO fields (trial setting+detail, pretrial conf, amended pleadings, discovery close, P/D expert designations, MSJ, expert challenges, mediation, witness/exhibit lists, limine), each with an editable ¶ cite, plus unlimited custom rows. **§ 18.001 is tracked explicitly (Michael 2026-07-13)** — a required selector asks how THIS order sets affidavit deadlines: 'same day as P expert designations' (typical; default — dockets a court-order 18.001 row at the expert date) / 'DIFFERENT date' (reveals its own date* + ¶ cite; the docketed row is flagged 'DIFFERENT from expert designations (<expert date>)' so the mismatch is visible) / 'order silent' (statutory per-defendant 90-day clocks stand, with the earlier-of pull-in if experts land sooner). A court-ordered 18.001 date REPLACES the statutory clocks (court order beats rule); 'diff' without a date is refused. Dates entered EXACTLY as written — no computation; backward-counted deadlines roll EARLIER, attorney-verify. Saving dockets everything as src=court_order, VACATES `supersedable` rule baselines (court order beats rule), updates the trial queue + court case card, and creates/updates the mediation card (an early/voluntary mediation converts to court-ordered). **Amended order** = warning + all prior court-order deadlines vacated and replaced; rule/notice/internal rows untouched. Signed chain shows 'Sep 02, 2025 (amended Apr 14, 2026)'.
- **Court-scoped local-rule deadline templates (Michael 2026-07-13)**: rule templates arm the day suit is filed, keyed to the SPECIFIC court assigned — not county-wide. **Bexar County: the 45-day DCO-filing rule applies to the 225th Judicial District ONLY (Michael's correction 2026-07-13; 2026 amendment, not in the published 2024 rules PDF; ATTORNEY-VERIFY the cite)** — a 225th case dockets 'DCO must be FILED' at petition+45d with a court-rules badge (amber 'DCO due <date>' → green '✓ satisfied'; entering the DCO vacates the row); other Bexar courts show 'no court-specific template (45-day rule = 225th only)'. **Bexar trial-setting expectation: cases set for trial within 18 months of APPEARANCE date (Michael 2026-07-13)** — this is the answer-entry trial-target row (RJA 6.1 / Bexar practice, DWOP watch, superseded by a DCO trial setting). **Webb County: calendar-call template intentionally NOT LOADED** — rule text unverified (webbcountytx.gov local-rules PDFs blocked from automated retrieval); Webb cases show a warning to calendar by hand until verified text is added. Principle: a template auto-dockets ONLY from verified rule text, scoped to the exact court it governs; anything unverified shows a warning, never a guess.
- **To Be Filed queue (Michael 2026-07-13)**: pre-suit matters assigned to this paralegal where suit is the next move, tightest limitations first. Row: client/DOI/county · SOL countdown (red <45d, row highlighted) · why we're filing · petition status (not started → drafting → ready for review → approved by Michael → FILED) · next action · '✓ FILED — enter it' (cause # + court + filing date; matter moves to My Cases with service tracking armed). Filing date must leave a SERVICE buffer before SOL. Minor/verify-flagged SOLs show ⚠ ATTORNEY-VERIFY, never a computed number.
- **SOL Watchlist (Michael 2026-07-13)**: every case assigned to this paralegal, riskiest first — UNFILED matters float to the top; **a filed case STAYS on the watchlist until the LAST defendant is served (Michael 2026-07-13)** — the row shows 'N of M defendants unserved; limitations only tolls with due diligence — the attempt log is the record'. Serving the final defendant moves the case to a muted '✓ Cleared — off watch' line (visible, expandable back into the case; returns to watch automatically if a new defendant is added unserved). Backed by core.compute_pi_sol; minors/WD/governmental-notice never trust the 2-year default.
- Work-queue tabs, each a DIFFERENT look (Michael): 
  - **Service Incomplete**: per unserved defendant — attempts, WHY, next step; near-SOL rows flagged (diligence = tolling record).
  - **Depositions**: scheduling → confirmed → READY only when readiness checklist complete (notice, reporter, videographer, exhibits, outline, witness confirm). Expandable checklist per depo.
  - **Written Discovery**: two sections — OUR responses due / THEIR deficient (pipeline: letter→conferral→MTC; will-supplement staleness clock).
  - **Mediations**: stage board per case (no DCO / selecting / availability out / dates received / scheduled) + deadline + next action.
  - **Trials**: soonest first, days-out badge, pretrial deadline summary, prep checklist count.
  - **Deadline Horizon**: 45d all cases, citations, JX + COURT ORDER tags.
- **Case timeline — pizza tracker (Michael 2026-07-13)**: every paralegal case page opens with a horizontal stage tree showing exactly where the case sits. 17 nodes: Suit filed (jury demand+fee) → Citation issued → Service (PER-DEFENDANT branch: ✓/⚠ + attempt counts) → Returns on file (TRCP 107(h) 10-day default clock — Michael's list was missing this) → Answers (per-defendant; 🔔 OVERDUE→default? branch) → Scheduling order (+ 225th 45-day rule — also missing) → Initial disclosures → Discovery served → Responses in → 10-day letter/MTC (deficiency pipeline) → § 18.001 affidavits → Experts (designations + challenges) → Depositions → Dispositive motions/MSJ (missing) → Mediation → Pretrial (lists·limine·docket call — missing) → Trial. Node states: ✓ done (muted) / ● now / ! warn / 🔔 crit / ○ future(dashed), with a 'YOU ARE HERE' tag on the first incomplete node. EVERY node derives live from the real record (service entries, answer entries, DCO fields, discovery statuses, deadline rows) — nothing is hand-set. Footer: settlement is an exit ramp at EVERY node → release → dismissal → liens & disbursement; blown answer deadline branches to the default-judgment track; TRCP 193.5 supplementation duty runs the whole way.
- **Switcher is two-way (Michael 2026-07-13)**: in CM view, a paralegal gets a persistent '🔁 Paralegal view' button in the TOP BAR (not just the banner link) to jump back; hidden for real CMs.
- Case page: court case card (190 level, jury, HB 19, DCO supersession), per-defendant boxes (served+answered collapse; unserved show diligence log + next step), discovery table, mediation card, deadlines, depos.
